# Sidervia 产品需求基线

- 状态：已确认，作为首版实现基线
- 版本：0.1
- 日期：2026-07-16
- 目标受众：项目维护者、实现者、审计者和部署者

## 1. 产品定义

Sidervia 是面向个人和小团队的自托管 AI API 网关与控制平面。系统将多个提供商、上游端点和凭证统一为受控入口，并提供管理页面、账号路由、协议兼容、状态资源绑定、用量记录和成本估算。

首个生产场景是单台 VPS、单管理员、不超过 5 个下游用户。架构应保持清晰和可扩展，但不得为了未来的集群或 SaaS 场景提前引入 PostgreSQL、Redis、消息队列或插件运行时。

## 2. 成功标准

项目达到首个可用版本时，必须同时满足：

1. 管理员能通过 Web 页面完成上游、账号、代理、模型路由和 Client Key 的配置。
2. 至少能以原生协议调用 OpenAI、Anthropic、Gemini 和 xAI 的文本/流式核心接口。
3. API Key 与至少一种官方公开 OAuth/device-code 流程能完整完成添加、刷新、失效和重新认证。
4. 相同公开模型可以绑定多个账号，调度过程可解释且不会自动切换到不同模型。
5. 强状态资源始终返回创建它的原账号；账号不可用时给出明确错误，而不是静默换号。
6. 每个请求都有不含正文的审计元数据、用量明细和可追溯的价格目录版本。
7. 单机 SQLite 部署在 2 vCPU、2 GiB RAM、40 GiB 磁盘上通过规定的功能、并发和恢复测试。
8. 安全发布门禁没有未处理的 Critical/High 问题，凭证、日志、代理和 OAuth 流程通过专项审计。

## 3. 用户与权限

### 3.1 管理员

系统只存在一个逻辑管理员。管理员可管理全部配置和查看全部元数据，不设计组织、角色、团队或细粒度 RBAC。

管理员必须使用密码登录，可选启用 TOTP。管理会话使用安全 Cookie，不允许把管理密钥存入浏览器 Local Storage。

### 3.2 下游客户端

下游客户端使用 Sidervia 生成的 Client Key 调用公开 API。Client Key 支持创建、命名、禁用、吊销和查看最后使用时间及用量，但 v1 不提供余额、硬配额、充值或支付。

Client Key 不是管理身份，不能访问 `/api/admin/*`。

## 4. 功能需求

### 4.1 管理页面

- **ADM-001**：提供响应式单页 Web 管理界面，由 Go 二进制内嵌并同源提供。
- **ADM-002**：提供总览页，展示账号状态、当前并发、请求趋势、错误率、估算成本和需重新认证事项。
- **ADM-003**：提供上游、账号、出站代理、模型路由、Client Key、用量成本、价格目录、系统设置和审计事件页面。
- **ADM-004**：所有高风险变更必须要求有效管理会话和 CSRF 校验，并产生审计事件。
- **ADM-005**：密钥和令牌创建后最多显示一次；后续页面只显示类型、前缀、到期时间和状态。
- **ADM-006**：UI 必须明确区分“健康”“冷却中”“额度受限”“凭证过期”“需要重新认证”和“管理员禁用”。

### 4.2 提供商与上游

- **UP-001**：首批内置 Provider 为 `openai`、`anthropic`、`google`、`xai` 和 `openai-compatible`。
- **UP-002**：Provider 是编译期适配器；v1 不加载第三方动态插件或执行外部脚本。
- **UP-003**：一个 Provider 可配置多个 Upstream；Upstream 包含基础 URL、协议选项、TLS/网络策略和默认出站代理。
- **UP-004**：自定义基础 URL 默认只允许 HTTPS，且必须经过 SSRF 防护。访问私网、回环或链路本地地址需要管理员显式开启。
- **UP-005**：系统可发现或手工登记模型及能力，但不得仅凭模型名称猜测能力后自动开放接口。
- **UP-006**：每个适配器都必须声明版本、原生能力、可转换能力、认证方式和已知限制。

### 4.3 账号与凭证

- **ACC-001**：账号支持 API Key、标准 OAuth Authorization Code + PKCE、OAuth Device Authorization Grant，以及提供商公开支持的工作负载身份方式。
- **ACC-002**：不接收账号密码、浏览器 Cookie、Session Key，也不读取或导入官方 CLI 的本地凭证文件。
- **ACC-003**：OAuth token exchange 和 refresh 只在服务端执行，前端永远不能读取 access token 或 refresh token。
- **ACC-004**：凭证使用必填的外部主密钥进行 AES-256-GCM 加密后写入 SQLite；缺少主密钥时服务拒绝启动。
- **ACC-005**：账号在通过轻量验证之前不可进入调度池。
- **ACC-006**：每个账号可覆盖绑定 HTTP、HTTPS 或 SOCKS5 出站代理。服务端 token exchange、刷新、额度探测和推理必须使用同一出站身份；标准 OAuth 的浏览器授权前台由管理员浏览器直接访问 IdP，必须在 UI 中明确这一网络边界。
- **ACC-007**：系统在到期前后台刷新，也在需要时按账号执行 singleflight 刷新；旋转后的 refresh token 必须原子替换。
- **ACC-008**：`invalid_grant` 或不可恢复的认证错误将账号置为 `reauth_required`，不得无限重试。
- **ACC-009**：401 最多触发一次刷新和重试，且仅限尚未向下游发送响应头或正文的请求。
- **ACC-010**：面向订阅产品的 OAuth 适配器必须标为 Beta，并明确提示使用者核对提供商条款；不得伪装官方客户端或依赖未公开接口。
- **ACC-011**：认证方式与计费类型必须分开建模。账号由管理员标记为 `subscription`、`metered` 或 `custom`，不能把 OAuth 自动视为订阅、把 API Key 自动视为按量。

### 4.4 模型路由与账号池

- **ROUTE-001**：公开模型 ID 默认保留上游原始模型名称。
- **ROUTE-002**：管理员可以明确创建同名模型组，将多个真正相同的上游模型作为候选；系统不得自动合并仅名称相似的模型。
- **ROUTE-003**：调度硬过滤条件包括：模型与能力匹配、账号启用、凭证有效、未处于硬冷却、并发槽可用、资源绑定一致。
- **ROUTE-004**：强状态 ID（如 `previous_response_id`、File、Batch、Video）优先且强制路由到原账号。
- **ROUTE-005**：普通会话支持可逃逸的软亲和；亲和账号不可用时可以在同模型候选内重新选择。
- **ROUTE-006**：过滤后只考虑最高优先级层。订阅账号默认高于按量 API Key，但管理员可设置为相同优先级。
- **ROUTE-007**：同优先级内选择当前负载率最低的账号，负载率相同则使用轮转游标打破平局。
- **ROUTE-008**：系统不得自动降级到不同模型，也不得把媒体、实时或批处理请求伪造成普通文本工作流。
- **ROUTE-009**：每个账号有独立最大并发和进程内信号量。v1 不使用 Redis 分布式锁或跨节点并发计数。
- **ROUTE-010**：401、429、5xx 和网络故障分别采用认证失效、官方重置时间和有抖动指数冷却；普通调用方 4xx 不惩罚账号。
- **ROUTE-011**：管理页面和请求诊断必须能解释候选为何被过滤及最终账号为何被选中，但不得泄露凭证。
- **ROUTE-012**：不含 `model` 的资源创建接口必须使用显式 Sidervia 路由提示或管理员配置的协议/端点默认路由；存在歧义时返回错误，不得猜测 Provider/账号。

### 4.5 协议与能力

- **PROTO-001**：能力策略为“原生全覆盖、共有语义转换”。Provider 原生能力通过原生协议暴露，共有能力才进入统一语义层。
- **PROTO-002**：跨协议共享范围限定为文本、图片/文档输入、流式输出、系统指令、工具调用、结构化输出、推理控制和用量信息。
- **PROTO-003**：音频、实时会话、图片/视频生成、Files、Batches、Embeddings 等非共有语义默认使用提供商原生协议。
- **PROTO-004**：无法可靠转换时返回稳定的 `capability_not_supported`，不得静默丢弃关键语义。
- **PROTO-005**：原生响应路径在安全边界内保留 Provider 新增字段和事件；原生请求路径不得盲目透传未知顶层或嵌套字段，也不得在应为标量的控制项中接受对象或数组。官方 Provider 只转发适配器已验证字段和文档明确的扩展容器；自定义 OpenAI-compatible Upstream 只能额外转发管理员显式允许的字段路径。所有路径都必须剥离下游认证、逐跳头、网关私有字段及客户端指纹字段。
- **PROTO-006**：转换路径使用带版本号的 Canonical IR，并记录所有已知降级项。
- **PROTO-007**：系统不伪造官方客户端名称、版本、TLS 指纹、状态回声或私有字段，也不承诺上游无法识别代理。
- **PROTO-008**：SSE、WebSocket 和媒体流必须提供背压、取消传播和有界缓冲，不能把完整响应读入内存后再返回。
- **PROTO-009**：一旦向下游发送任何响应字节，系统不得透明换号重试。

### 4.6 状态资源

- **RES-001**：Files、Batches、Responses、Videos 等资源创建后生成 Sidervia 不透明 ID，并持久化 Provider、Upstream、Account 和上游资源 ID 映射。
- **RES-002**：后续读取、删除、取消或继续操作必须沿映射返回原账号。
- **RES-003**：绑定账号不可用时返回可识别的 `bound_account_unavailable`、`reauth_required` 或上游错误，不得跨账号回退。
- **RES-004**：需要时在 JSON 和 SSE 中重写资源 ID，但不得修改未知业务字段。
- **RES-005**：资源映射具有类型、状态、上游有效期和本地清理时间，清理操作不应误删仍有效的上游资源。
- **RES-006**：资源映射绑定创建它的 Client Key。其他 Client Key 即使获得资源 ID 也不能读取、继续、取消或删除；管理员重新分配必须是显式且有审计的操作。

### 4.7 文件与媒体

- **MEDIA-001**：上传、下载、图片、音频和视频优先端到端流式转发。
- **MEDIA-002**：系统不长期归档提示词、响应正文、文件或媒体内容。
- **MEDIA-003**：只有上游协议要求可重放、已知长度或 seek 时才允许临时落盘；请求结束立即删除，崩溃残留最迟 24 小时清理。
- **MEDIA-004**：临时目录必须有单文件限制、全局容量限制和最小剩余磁盘保护。
- **MEDIA-005**：数据库只保留资源映射、内容类型、大小、状态、有效期和计费用元数据，不保留媒体本体。

### 4.8 用量与成本

- **USAGE-001**：每次请求记录 Client Key、公开模型、实际 Provider/Upstream/Account、协议、状态、延迟、TTFT、流式标记、错误分类和用量，不记录正文。
- **USAGE-002**：原始请求明细保留 365 天；日/月聚合永久保留，管理员可手动导出后清理。
- **USAGE-003**：成本模型支持 token、缓存 token、推理 token、工具调用、图片规格、音频时长/字符、视频时长、Embedding、Batch 折扣、优先级倍数和文件存储等计量单位。
- **USAGE-004**：每条成本记录绑定价格目录版本和生效区间。历史成本不得因编辑当前价格而变化。
- **USAGE-005**：提供商返回官方且语义明确的实际费用时优先使用；否则使用管理员审核过的官方价格目录。
- **USAGE-006**：订阅账号展示“按官方 API 价格计算的等价值”，不把月费摊销成虚假的单次成本。
- **USAGE-007**：自定义 OpenAI-compatible 上游的价格由管理员配置并明确标为自定义来源。
- **USAGE-008**：价格目录变更必须记录来源 URL、审核人、审核时间、币种和内容摘要哈希。

### 4.9 运维与可观测性

- **OPS-001**：提供 `/healthz`、`/readyz` 和仅内网可访问的 Prometheus `/metrics`。
- **OPS-002**：日志使用结构化格式，默认脱敏 Authorization、Cookie、API Key、OAuth code/token、代理凭证和请求正文。
- **OPS-003**：服务必须处理 SIGTERM：停止接收新请求、等待有界时间、刷新用量批次并关闭数据库。
- **OPS-004**：SQLite 使用 WAL、busy timeout 和短事务；同一数据目录只允许一个 Sidervia 实例持有进程锁。
- **OPS-005**：提供一致性备份与恢复流程，数据库备份和主密钥必须分开保存。
- **OPS-006**：镜像以非 root 用户运行，提供 amd64/arm64 版本并发布 SBOM、校验和与镜像签名信息。
- **OPS-007**：升级必须先备份，迁移必须事务化；不可逆迁移需要明确版本门禁和回滚说明。

## 5. 非功能需求

### 5.1 性能基线

在 2 vCPU、2 GiB RAM、正式构建且使用本地 SQLite 的目标机器上：

- 20 个并发流式连接持续 10 分钟，不发生 OOM、死锁或数据竞争。
- 不含上游耗时的普通代理处理开销 p95 不高于 50 ms。
- 不含上游首包耗时的 SSE 转发附加 TTFT p95 不高于 50 ms。
- 稳态 RSS 目标不高于 512 MiB；临时媒体缓冲和操作系统页缓存单独计量。
- 10,000 条请求明细写入期间，管理查询不会持续阻塞代理请求。

这些是发布验收基线，不是对互联网端到端延迟的承诺。

### 5.2 可靠性

- 进程崩溃重启后，已提交的凭证、资源绑定、价格版本和重要冷却状态必须保持一致。
- 用量异步写入允许进程被强杀时丢失最多一个短批次，目标上限为 2 秒或 100 条记录，以先到者为准。
- 上游超时、下游取消和代理断开不得泄漏 goroutine、文件描述符、临时文件或并发槽。

### 5.3 安全与隐私

- 默认拒绝不安全配置，任何私网访问、弱 TLS 或调试正文日志都必须显式开启并显示风险。
- 密钥、凭证和 TOTP secret 在数据库中加密；Client Key 只保存不可逆验证值。
- 管理端只允许同源访问，默认关闭 CORS，所有状态变更需要 CSRF 防护。
- 不将提示词、输出正文、工具参数或媒体内容写入数据库、日志、指标或错误报告。
- 安全要求及验证证据见[安全审计方案](security-audit.md)。

### 5.4 可维护性

- Go 后端和 React 前端使用单仓库，生产发布为单二进制和单容器。
- Provider 特性通过小而明确的接口隔离，不在全局处理链堆叠提供商条件分支。
- 数据库迁移只前进且有测试；Schema、API 和 Canonical IR 都有显式版本。
- 新能力必须先更新能力矩阵和协议契约，再增加实现。

## 6. v1 明确不做

- 面向公众注册的用户体系、组织、角色和租户隔离。
- 充值、余额、支付、订单、发票和盈利结算。
- PostgreSQL、Redis、Kafka、分布式调度和多副本共享 SQLite。
- 自动切换到价格更低、能力相似或名称相近的其他模型。
- 本地长期保存提示词、响应或媒体内容。
- 通用工作流/Agent 编排平台和跨模态伪转换。
- 动态 Provider 插件、任意脚本钩子或远程代码执行扩展点。
- CLIProxyAPI/Sub2API 的配置、管理 API、数据库、用户或 UI 兼容层。
- 规避提供商的客户端识别、服务条款、额度限制或安全控制。

## 7. 发布验收

首个可用版本必须通过以下端到端场景：

1. 全新启动、设置管理员密码、启用 TOTP、退出并重新登录。
2. 添加 API Key 账号和 OAuth 账号，验证、刷新、模拟 token rotation 与重新认证。
3. 为同一模型配置两个账号，验证优先级、负载选择、并发限制、429 reset 和 5xx 冷却。
4. 用 Client Key 分别执行非流式、SSE 和至少一种原生 WebSocket 请求。
5. 创建强状态资源，重启 Sidervia 后继续访问仍命中原账号。
6. 通过 HTTP/HTTPS/SOCKS5 账号代理完成登录、刷新、探测和推理的一致出站验证。
7. 验证数据库、日志、指标、审计导出和错误页面中均不存在请求正文与明文凭证。
8. 导入两个价格目录版本，确认历史请求成本不被重算覆盖。
9. 在 40 GiB 磁盘预算下转发大文件并模拟崩溃，确认临时文件清理和磁盘保护生效。
10. 完成备份、删除数据目录、恢复数据库与主密钥并通过就绪检查。

详细测试矩阵见[测试方案](testing.md)。

## 8. 假设与风险

- Provider API、模型、价格和 OAuth 政策持续变化；适配器必须声明其验证日期，价格必须版本化。
- 订阅产品登录成功不等于允许第三方池化转发。实现和部署者必须分别核对适用条款，Sidervia 不提供规避机制。
- 单机是有意的产品约束。需要多副本、高可用或大量用户时，应先重新设计存储和一致性，而不是把 SQLite 放到共享网络盘。
- 对媒体和实时连接而言，VPS 带宽、连接数和上游限流通常比 CPU 更早成为瓶颈。
